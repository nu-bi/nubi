# Docs & screenshots

How Nubi's documentation works, how to add or change a doc page, and how the
automated screenshot pipeline keeps every image in the docs and README in sync
with the product.

## How docs work

Docs are plain markdown files in `docs/`. The same files serve two audiences:

1. **In-app** — rendered at `/docs/<slug>` by `src/pages/DocsPage.jsx`. The
   slug is the filename without `.md` (`docs/git-sync.md` → `/docs/git-sync`;
   `index.md` → `/docs/home`).
2. **On GitHub** — browsable as regular markdown.

The in-app navigation is **explicitly curated** in `src/docs/registry.js`: a
file in `docs/` is only reachable in-app if its path is listed in the
`import.meta.glob` array *and* its slug appears in `LAYOUT` (which controls
section/group/order). Internal design documents (e.g. `flows-v2-design.md`,
`ingestion-design.md`) deliberately stay unregistered — they live in the repo
for contributors but are not part of the product docs.

### Adding a doc page

1. Create `docs/my-page.md`. The first `# H1` becomes the nav title.
2. In `src/docs/registry.js`, add `'/docs/my-page.md'` to the glob array and
   the slug `'my-page'` to the right `LAYOUT` group.
3. Link to it from related docs.

### Authoring conventions

- **Internal links** must be root-prefixed slugs: `[Flows](/docs/flows)`.
  `MarkdownRenderer` turns `/`-prefixed hrefs into client-side router links;
  relative links (`./flows.md`) render as plain anchors and 404 in-app.
- **Screenshots**: `![Alt text](/docs/screenshots/<name>.png)`. Only reference
  images that the screenshot pipeline generates (see below) — the pipeline
  fails if a doc references an image it no longer captures.
- **Illustrations**: `![Alt](illustration:Name)` is special syntax resolved to
  an SVG component via `src/components/illustrations/docMap.js`. Leave these
  alone unless you are also adding the component.
- **EE features** (billing, licensing — anything under `backend/app/ee/`)
  must be marked as enterprise-only, GitLab CE/EE style.
- Don't document planned features. Docs describe what is shipped; designs live
  in unregistered `*-design.md` files.

## The screenshot pipeline

Every product screenshot in the docs **and** the README hero images are
generated by one script — never hand-captured — so they can be regenerated to
match whatever version the code is at.

```bash
npm run screenshots        # against an already-running dev stack (:5173/:8000)
npm run screenshots:auto   # self-contained: boots an ephemeral seeded stack,
                           # captures, tears down (needs docker)
```

`screenshots:auto` reuses `scripts/e2e.sh` (via its `E2E_RUN_CMD` override),
so the capture always runs against a clean, freshly seeded database.

### What it produces

| Output | Contents |
|--------|----------|
| `public/docs/screenshots/*.png` | every screenshot referenced from `docs/*.md` (1440×900 @2x, light theme) |
| `docs/assets/hero-{light,dark}.png` | the README hero — the demo dashboard in full-screen view, one per color scheme |
| `public/docs/screenshots/manifest.json` | version, git commit, timestamp, and file list for the run — makes stale screenshots detectable |

### Guarantees

- **Self-sufficient**: resource IDs (boards, queries, flows) are discovered by
  name through the API on every run; if the seed lacks a demo flow, the script
  creates the "Retail revenue pipeline" example itself. Reseeding never breaks
  it.
- **Stale-reference check**: after capturing, the script scans `README.md` and
  `docs/*.md` for image references and **exits non-zero** if any referenced
  image was not regenerated in that run. Renaming or dropping a capture
  without updating the docs fails the run — wire it into CI as a docs-drift
  gate.
- **Deterministic look**: light theme is forced, viewport is fixed, and the
  flows captures pin the `dev` environment (where unpromoted drafts live).

### Adding a screenshot

In `scripts/docs-screenshots.mjs`:

1. Add a `goAndShoot(page, '/route', 'my-name', { settle: 1500 })` call (or a
   custom interaction block — see the flows section for view-toggling
   examples).
2. Reference it from a doc as `/docs/screenshots/<my-name>.png` (base name —
   the dark variant is picked automatically in dark mode).
3. Run `npm run screenshots` and eyeball the PNG before committing.

Navigation guidance for new captures: the script navigates **client-side**
(`ensureApp`) and boots through `/home` on full reloads. Don't replace this
with raw `page.goto(deepLink)` chains — full page loads rotate the refresh
cookie, and rapid successive boots can trip the backend's token-reuse
detection and log the session out mid-run; cold deep-links can also mount
pages before auth is ready, capturing empty states.

### When to regenerate

Regenerate (and commit the changed PNGs) whenever a PR visibly changes UI that
appears in the docs — layout, navigation, major styling — or adds a page worth
documenting. The manifest's `version`/`commit` fields tell you exactly what
state the current images came from.
